### 1.1 项目背景

作为一名社会学专业的独立开发者，我在日常学习和工作中积累了较丰富的 Python 开发经验，但前端方面仅掌握基础知识。在开发各种桌面小工具（数据格式转换、批量处理、模型推理辅助等）时，我始终面临一个核心矛盾：Python 后端能力强大，但图形界面方案要么界面简陋 (tkinter)、要么学习成本过高 (PyQt)、要么需要引入一整套额外的后端 (Electron + Flask/FastAPI)。

我需要一个能在**编程开发效率/便利性**与**全栈框架成熟度/效果**之间取得平衡的方案。pywebview 用系统自带的 WebView 渲染前端、Python 做后端，天然前后端分离，这个思路很对 -- 但实际使用时，前后端通信的样板代码、开发环境搭建、生产打包等环节仍然有大量重复劳动。于是我决定自己造轮子：构建一个"克隆即开发"的框架，让像我这样主要熟悉 Python 的开发者只需关注业务逻辑，前后端通信、开发热重载、打包构建全部开箱即用。

### 1.2 用户画像

- **主要用户**: 和我类似背景的独立开发者 -- 熟悉 Python 但前端经验有限，需要在不深入学习前端工程化的前提下构建有现代感的桌面应用

## 2. Pain Points & Motivation

### Before (痛点)

- **前后端通信样板代码多**: 使用 pywebview 时，每次都需要手动处理 JS 调用 Python 方法的绑定、异常捕获、响应格式统一，一个简单的方法暴露需要写 10+ 行样板代码
- **开发环境搭建繁琐**: 前端 Vite 热重载 + 后端 pywebview 窗口需要分别启动，端口管理、进程清理等操作对不熟悉前端的同学来说门槛高
- **生产打包配置复杂**: PyInstaller 配置涉及前端资源嵌入、平台差异处理、DLL 冲突等问题，新手往往在打包环节耗费大量时间
- **跨平台线程安全问题**: Windows 下 WebView2 的 COM 线程模型要求 `evaluate_js` 必须在主线程调用，但 pywebview 原生未提供从后台线程安全推送事件到前端的机制

### After (改进效果)

- **零样板代码**: 开发者只需继承 `Bridge` 类并用 `@expose` 装饰器标注方法，框架自动处理异常捕获和响应格式统一，一个方法暴露仅需 3 行代码
- **一键开发启动**: `dev.bat` / `dev.sh` 一键完成依赖安装 + Vite 启动 + 应用窗口打开，开发环境搭建从 30 分钟缩短到 1 分钟内
- **开箱即用的打包方案**: 内置 PyInstaller 配置模板 (`app.spec`) 和自动构建脚本 (`build.py`)，支持 onedir / onefile / Android 三种模式
- **线程安全的事件系统**: 通过事件队列 + JS 定时器轮询的方式，实现了从任意线程安全推送事件到前端的功能

## 4. Product Solution

### 4.1 方案概述

PyWebVue 的核心价值：**极简的 Python + Vue 双向通信框架，克隆即开发**。框架核心仅 3 个文件约 200 行 Python 代码，提供：

1. **Python -> JS**: `@expose` 装饰器自动暴露方法，统一 `{"success": bool, "data": ...}` 响应格式
2. **JS -> Python**: `Bridge._emit()` 线程安全推送事件，前端通过 `onEvent()` 监听
3. **开发工具链**: 一键开发脚本 (dev.py)、自动构建脚本 (build.py)、PyInstaller 配置模板 (app.spec)

### 4.2 核心功能设计

#### 功能 1: @expose 装饰器 + 统一响应格式

- **做什么**: 开发者在 `Bridge` 子类中用 `@expose` 装饰器标注方法，框架自动包装 try/except 异常处理，所有方法返回统一的 `{"success": bool, "data": ..., "error": ...}` 格式
- **为什么这样设计**: 解决前后端通信中异常处理不统一的问题。作为非科班开发者，我希望尽可能减少前端侧的错误处理负担，让 Python 端的错误信息能以统一格式传递到前端，前端只需要检查 `success` 字段即可
- **设计取舍**: 考虑过使用 Result 类型 (Rust 风格) 但在 Python 中实现不够自然，最终选择了最简单的 dict 格式，与 pywebview 的 JS 互操作最兼容

#### 功能 2: 线程安全事件队列系统

- **做什么**: 后台线程通过 `_emit()` 将事件放入队列，JS 定时器每 50ms 调用 `_tick()` 在主线程刷新队列并通过 `evaluate_js` 分发 CustomEvent
- **为什么这样设计**: Windows 下 WebView2 基于 COM 线程模型，`evaluate_js` 必须在主线程调用。直接从后台线程调用会导致进程崩溃。这是在后续另一个项目中集成 ONNX Runtime + WebView2 时实际遇到的
- **设计取舍**: 考虑过使用 `webview.windows[0].evaluate_js` 的线程代理方案，但 pywebview 本身不提供线程安全的 JS 执行接口。最终选择了队列 + 轮询的经典模式，延迟可控 (50ms) 且实现简单

#### 功能 3: 主线程任务执行机制

- **做什么**: 提供 `register_handler(name, handler)` + `run_on_main_thread(name, args)` API，让后台线程可以安全地将 C++ 扩展初始化等操作调度到主线程执行
- **为什么这样设计**: 某些 C++ 扩展 (如 ONNX Runtime、sherpa-onnx) 必须在创建 WebView2 的主线程上初始化，否则会触发 DLL 冲突或 COM 违规崩溃
- **设计取舍**: 考虑过子类继承重写的方式但扩展性差，最终选择了命令注册模式 (Registry Pattern)，支持动态注册多个处理器，无需继承

#### 功能 4: 一键开发与打包工具链

- **做什么**: `dev.py` 自动完成 uv sync + npm install + Vite 启动 + 应用启动；`build.py` 自动完成前端构建 + PyInstaller 打包
- **为什么这样设计**: 作为非科班开发者，我最能体会"环境搭不起来"和"打包不会配"的痛苦。一键脚本消除了这两个最常见的阻塞点
- **设计取舍**: 没有选择 CLI 脚手架 (如 create-react-app) 方案，因为"克隆即开发"更简单直接，用户只需 `git clone` 就能开始

### 4.3 系统架构

```
+------------------+     +--------------------+     +------------------+
|   Vue 3 前端     |     |   pywebview        |     |   Python 后端    |
|   (TypeScript)   |     |   WebView 窗口     |     |                  |
|                  |     |                    |     |                  |
| call() ----------+---->| window.pywebview   +---->| @expose 方法     |
|                  |     |   .api.method()    |     |   返回 JSON      |
| onEvent() <------+-----+ dispatchEvent()   <-----+ _emit() 推送     |
|                  |     | CustomEvent        |     |   (线程安全队列) |
+------------------+     +--------------------+     +------------------+
```

- **前端**: Vue 3 + Composition API (`<script setup>`) + TypeScript，Vite 6 构建工具
- **后端**: Python 3.10+，pywebview 6.0 作为桌面容器
- **通信**: `window.pywebview.api` (JS -> Python) + `CustomEvent` (Python -> JS)
- **打包**: PyInstaller (桌面) + Buildozer (Android)

## 5. Development Process

### 5.1 规划与调研阶段 (2026-03-28 上午)

**问题发现**: 作为一名社会学专业的独立开发者，我在使用 Python 开发各类桌面工具的过程中，深刻感受到"Python 后端能力强但 GUI 方案薄弱"的核心矛盾。确认了自己和周围有类似需求的朋友在工具开发效率方面的实际需求。

**技术调研**: 对比了 Electron、Tauri、pywebview 三个方案。Electron 需要独立的 Node.js 后端与 Python 后端并存，架构过重；Tauri 使用 Rust 后端，与 Python 生态不兼容；pywebview 直接使用系统 WebView，Python 即后端，最符合"Python 开发者快速构建桌面应用"的需求。

**MVP 范围**: 确定第一版实现三个核心功能 -- @expose 方法暴露、事件推送、一键开发启动。

### 5.2 第一版实现 -- 重型架构 (2026-03-28 下午)

在一天内完成了第一版架构，包含 12 个 Python 模块 (约 2,000 行核心代码 + 8,000 行模板/示例)，包括：

- 完整的 API 基类、配置系统 (YAML)、事件总线、日志系统、单例模式、结果类型、进程管理、原生对话框封装
- CLI 脚手架系统，支持通过命令行生成新项目
- 两个完整示例应用 (文件工具、进程管理工具)

这一版的思路是"功能完备"，参考了传统 Web 框架的设计模式。当天完成开发后，编写了完整的用户指南和 AI 辅助开发文档。

### 5.3 架构重构 -- 从"完备"到"极简" (2026-03-29 ~ 03-30)

**重构触发**: 第一版完成后自己试用，并向周围有类似需求的朋友推荐后发现了几个关键问题：
1. 朋友反馈"模块太多，不知道从哪看起" -- singleton、result 类型等抽象层增加了理解成本，对非科班开发者尤其不友好
2. CLI 脚手架实际上不如"直接克隆仓库"来得直接
3. 配置文件 (config.yaml) 增加了一个不必要的抽象层，构造函数参数更直观

**重构决策**: 彻底简化。commit `925e1b3` 一次删除 13,569 行、新增 2,166 行：

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| 核心文件数 | 12 个 .py | 3 个 .py |
| 核心代码量 | ~2,000 行 | ~200 行 |
| 配置方式 | config.yaml | 构造函数参数 |
| 项目初始化 | CLI 脚手架 | git clone |

重构后的核心仅保留 `app.py` (窗口管理) + `bridge.py` (通信桥接) + `__init__.py` (导出)，新增了 `dev.py` (一键开发) 和 `build.py` (一键打包)。

**重构结果**: 朋友反馈新版本"一看就懂"，从克隆到运行出第一个 Demo 不到 5 分钟。

### 5.4 线程安全增强 (2026-04-07 ~ 04-12)

**触发**: 在另一个项目中使用 PyWebVue 集成 ONNX Runtime 时，发现后台线程调用 `_emit()` 在 Windows 上会导致 WebView2 崩溃。同时，C++ 扩展的主线程初始化需求也缺乏框架级支持。

**实现**: 在 `bridge.py` 中新增事件队列 (`queue.Queue`) + JS 定时器轮询 (`_tick`) 的线程安全事件推送机制，以及基于命令注册模式的主线程任务执行系统 (`register_handler` + `run_on_main_thread`)。`bridge.py` 从 71 行增长到约 200 行，但保持了对现有 API 的完全向后兼容。

**验证**: 代码审查发现了 12 个问题 (竞态条件、类型注解缺失等)，逐一修复后通过实际 ONNX Runtime 集成场景验证了稳定性。

## 7. Highlights & Challenges

### 7.1 技术挑战

- **挑战 1: Windows WebView2 COM 线程模型限制**
  - **问题**: WebView2 基于 Chromium，其 COM 模型要求所有 UI 相关调用 (包括 `evaluate_js`) 必须在主线程执行。当后台线程直接调用 `_emit()` -> `evaluate_js()` 时，Windows 上会触发访问违规崩溃
  - **解决方案**: 实现了事件队列 + JS 定时器轮询的异步通信机制。后台线程将事件放入 `queue.Queue`，JS 端每 50ms 调用 `_tick()` 在主线程刷新队列，通过 `document.dispatchEvent(new CustomEvent(...))` 分发到前端
  - **收获**: 深入理解了 Windows COM 线程模型和 WebView2 的底层机制，积累了跨平台 GUI 框架的线程安全设计经验

- **挑战 2: ONNX Runtime DLL 冲突**
  - **问题**: ONNX Runtime 的 C++ 扩展与 WebView2 的 Chromium 共享 MSVC runtime，DLL 加载顺序不当会导致版本冲突崩溃
  - **解决方案**: 提供了 `on_start` 回调钩子，允许用户在 WebView2 初始化前预加载原生库。同时在文档中明确了 Windows 下 C++ 扩展的集成最佳实践
  - **收获**: 理解了 Windows DLL 搜索顺序和动态链接库冲突的排查方法

- **挑战 3: 从"功能完备"到"极简"的架构重构决策**
  - **问题**: 第一版架构包含 12 个模块，功能全面但使用复杂，同事反馈学习成本高
  - **解决方案**: 基于"用户实际需要什么而非框架应该有什么"的原则，大胆删除了 90% 的代码，保留 3 个核心文件。CLI 脚手架改为"克隆即用"，YAML 配置改为构造函数参数。这对于非科班开发者尤其重要 -- 学习成本越低，框架才越有用
  - **收获**: 学会了"先做出来再用起来再决定保留什么"的迭代思路，避免过度设计

### 7.2 设计亮点

- **统一响应格式 (`@expose` + `ApiResponse<T>`)**
  - Python 端的 `@expose` 装饰器自动包装 try/except，TypeScript 端的 `call<T>()` 返回类型安全的 `ApiResponse<T>`，前后端错误处理完全一致

- **命令注册模式 (Registry Pattern) 替代继承**
  - `register_handler(name, handler)` + `run_on_main_thread(name, args)` 的设计比子类继承更灵活，支持运行时动态注册，无需修改类层次

- **dev/prod 自动切换**
  - 通过 `sys.frozen` 自动检测运行环境，开发时连接 Vite 热重载，生产时加载构建产物，用户无需手动切换

### 关键收获

- **简单优于完备**: 200 行的极简框架比 2,000 行的"功能完备"框架更受用户欢迎。做减法的判断力比做加法的技术能力更重要
- **实际使用是最好的设计评审**: 再多的提前规划都不如让真实用户试用一周来得有效。快速出 MVP -> 收集反馈 -> 迭代改进的工作流远优于"一次设计到位"
- **非科班也能做全栈**: 关键不是掌握多少技术栈，而是找到合适的平衡点 -- pywebview + Vue 让我能在 Python 熟悉区工作的同时获得现代化的前端效果，AI 辅助进一步弥合了经验差距
- **跨平台 GUI 开发的线程陷阱**: Windows/macOS/Linux 的 GUI 线程模型各不相同，框架层面的线程安全封装是必须的而非可选的
